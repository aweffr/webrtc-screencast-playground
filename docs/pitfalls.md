# 项目常见陷阱与验证规则

本文面向维护 macOS 发送端、WebRTC 构建产物、Android TV 接收端和端到端实验的开发者。它记录本项目已经实际发生、并且可能再次影响业务结论的失败模式。

项目的核心目标是会议场景下低延迟、高清晰度的极速投屏。规则因此只约束会影响以下判断的工作：文字是否清楚，正文更新是否及时，静态与动态内容切换是否正确，是否出现丢帧或卡顿，以及实验结论能否被证据复核。验证规模应与风险相匹配；不能减少必要的测试和端到端验证，也不应为了罕见边界增加与业务目标无关的防御代码或无限扩张实验组合。

## 与根目录规则的边界

根目录 [`AGENTS.md`](../AGENTS.md) 已经规定三项基础要求：跨平台共享 patch 必须验证每个目标平台的完整 patch chain；配置格式必须通过真实 parser 的解析规则测试；更换 WebRTC framework 后必须隔离旧 build cache。本文不重复展开这三项要求，而是补充它们尚未覆盖的问题：怎样证明测试确实经过目标路径，怎样证明指定构建产物确实进入最终程序，以及怎样设计、执行和解释端到端实验。

三仓的责任边界沿用 [`AGENTS.md`](../AGENTS.md) 中的依赖关系。规则应放在能够执行它的仓库：patch、parser 和 WebRTC 构建验证归 `../my-webrtc-builds`；本项目负责锁定并核对消费的构建产物，以及验证真实发送端和接收端行为。跨仓库事故可以在两边保留索引，但每个仓库的完成标准都应能在本仓库直接执行，不能只留下另一仓库才能落实的空泛要求。

每条规则都包含适用情况、已经发生的问题、后续规则和完成标准。文中的“完整链路冒烟验证”是指用一轮短流程确认发送、传输、接收、截图和分析全部可运行；它不能替代正式端到端实验。

## P01　把实现事实、底层能力和产品意图分开

### 适用情况

发现某个功能只对部分采集源、设备、codec 或运行模式生效，或者准备扩大、缩小其启用范围时。

### 已经发生的问题

静态内容感知（static-aware）MaxQP 曾经只对主屏镜像生效。这只能证明当时的实现范围，却一度被解释成“产品有意只支持主屏”。实际检查表明，虚拟屏和 media baseline 采集模式也可以使用同一机制，之后已补齐对应范围。错误不在于提出风险假设，而在于把没有历史依据的假设写成既定产品决策。

### 后续规则

先分别记录四类信息：

1. 当前实现如何判断是否生效；
2. 底层接口和目标设备是否具备能力；
3. ADR、历史提交或明确业务决策记录了什么原因；
4. 哪些内容仍只是待验证的工程假设。

这里的“功能启用范围”是指决定某项功能对哪些采集源、设备或运行模式生效的条件。不能仅凭现有条件分支反推产品意图。查不到第三类依据时，应明确写“原因未记录”，再通过 `git log`、`git blame`、设计文档和需求记录核实。

### 完成标准

- 变更说明能区分实现事实、能力事实、已记录的产品意图和待验证假设。
- 没有业务依据的旧限制不会因为“当前就是这样”而被继续保留。
- 扩大范围时采用最小可行改动，并按 P04 补充与新增风险相称的验证。
- 仓库内不再残留与新范围矛盾的说明、运行手册或默认配置。

### 项目证据

[`ScreenCaptureSource.swift`](../apps/macos/WebRTCScreencast/Capture/ScreenCaptureSource.swift) 及提交 `c2242f9` 记录了 static-aware MaxQP 启用范围的调整。

## P02　测试通过之前，先证明测试经过了目标路径

### 适用情况

准备使用单元测试、patch 检查、构建测试或完整测试套件，证明某个 parser、patch、构建产物或运行时行为正确时。

### 已经发生的问题

项目曾出现过两类“命令成功但目标行为未被覆盖”的情况：patch 检查只包含部分文件，没有经过各平台真实的完整应用顺序；配置测试编译并通过，但没有调用真正的 JSON parser，非法字段直到后续原生验证才暴露。测试数量和套件名称都不能弥补覆盖路径缺失。

### 后续规则

在声称验证通过前，写清以下映射：

```text
待验证行为 → 实际生产代码路径 → 覆盖它的测试目标 → 实际执行命令
```

路径不明显时，先引入一个“已知错误对照”：临时保留一个确定会触发目标错误的输入，确认同一条命令能够稳定失败，再恢复正确实现并确认通过。这个方法用来验证测试灵敏度，不需要长期保留故意错误。

测试应穿过最接近生产环境的最小边界，例如真实 parser、真实 patch chain、真实产物安装脚本或运行时读取接口。不要新增只能证明某段文字存在、某个私有字段被赋值的低价值断言。

### 完成标准

- 验证记录能从业务行为追溯到生产路径、测试目标和命令。
- 被证明的目标行为本身没有被 mock、条件编译、文件过滤或错误的 test target 绕开；与该行为无关的依赖边界仍可按测试目的使用 mock。
- 对高风险或不透明路径，已知错误对照能够证明测试会在回归时失败。
- 报告只陈述本次命令实际覆盖的范围，不用“全量测试通过”替代覆盖说明。

### 项目证据

根目录 [`AGENTS.md`](../AGENTS.md) 中的 patch 和 parser 规则来自这两类事故；本规则补充的是验证覆盖路径的方法。相关修复见 `../my-webrtc-builds` 的 `7c029bb`、`a8a3493`。

## P03　指定构建产物必须贯穿到最终可执行程序

### 适用情况

实验或构建命令允许指定 XCFramework、AAR、archive、commit 或产物目录时。

### 已经发生的问题

实验执行脚本曾把显式指定的 macOS 构建产物写入记录文件，但后续项目生成又调用默认 bootstrap 路径，用另一份 framework 覆盖了 `Vendor/`。记录文件看似正确，最终程序却没有使用指定二进制。清理 `DerivedData` 只能消除旧 module cache，无法修复这种产物流向错误。

### 后续规则

把产物身份沿真实数据流追踪到底：

```text
命令行输入 → 安装或解压 → bootstrap/project generation → 编译链接 → 最终程序 → 运行时
```

每一层优先记录不可逆 hash，并检查关键 header、symbol 或版本信息。默认路径不得在后续步骤静默覆盖显式输入；如果确实允许回退，必须在运行前明确输出最终选中的产物。

### 完成标准

- 输入 archive、安装后的 framework/AAR、最终 executable/APK 都有可核对的身份信息。
- 运行时遥测（telemetry）中的 codec、encoder/decoder 标识和关键能力与预期相符；这些信息作为 hash 链的运行时补充，不能代替产物身份校验。
- 产物替换后按 [`AGENTS.md`](../AGENTS.md) 清理或隔离旧 cache。
- 删除、改名或替换显式产物后，构建会失败或明确报错，不会静默改用默认版本。

### 项目证据

[`scripts/bootstrap-webrtc.sh`](../scripts/bootstrap-webrtc.sh)、[`scripts/run-hevc-meeting-experiment.sh`](../scripts/run-hevc-meeting-experiment.sh) 和提交 `2eff358` 记录了产物流向修复。

## P04　扩大 static-aware 编码策略的启用范围时，用两层验证保护真实风险

### 适用情况

静态内容感知 MaxQP、IDR 或同类编码策略从主屏扩到虚拟屏、从一种采集源扩到另一种采集源时。

### 已经发生的问题

static-aware MaxQP 扩展到虚拟屏和 media baseline 后，单元测试只保护了功能启用范围的枚举判断；它没有证明新采集源上会真实完成 `STATIC → ACTIVE → STATIC`，也没有证明 QP 更新、IDR 和 Android 接收端渲染正常。已有的主屏实验不能填补这项缺口。

### 后续规则

这类范围变更采用两层验证：

1. 单元测试保护“哪些来源应该启用”的公开行为要求；
2. 对每类新增风险至少跑一轮有界端到端流程，验证状态往返、参数实际应用和接收端可见结果。

有界验证不是重跑所有 codec 和参数组合。虚拟屏范围变更通常只需要一轮固定内容的 `STATIC → ACTIVE → STATIC`、QP/IDR 运行遥测和 Android renderer 检查；只有这一轮暴露差异时，才扩大实验。其他类型的功能扩展应验证各自新增的真实行为和风险，不能机械套用本条的 QP/IDR 验收项。

### 完成标准

- 功能启用范围的单元测试覆盖新增和保留的来源。
- 至少一轮目标来源的真实端到端流程完成，且状态、QP、IDR 和接收端画面均可核对。
- 仓库级搜索确认没有旧文档、默认参数或脚本继续排除新来源。
- 验证规模由新增风险决定，没有把范围修改扩张成无关的完整实验矩阵。

### 项目证据

提交 `c2242f9` 保护了范围判断；本规则明确其后仍需补充的目标来源端到端验证。

## P05　VideoToolbox 能力必须在准确的会话条件下实测

### 适用情况

准备启用 MaxQP、Spatial Adaptive QP、Low-Latency Rate Control、B-frame 或其他 VideoToolbox 属性时。

### 已经发生的问题

普通 HEVC 硬件编码会话支持的属性，不能直接外推到启用 Apple Low-Latency Rate Control 的 RTVC 会话。项目实测中，普通 HEVC 会话可以使用 `MaxAllowedFrameQP`，而 RTVC 会话的属性集合不同。只根据 SDK 声明、另一台机器或另一种会话得出的结论，会把“不支持”误判成代码错误，也可能把无效实验组当成有效配置。

### 后续规则

能力记录至少绑定以下条件：硬件型号、OS 版本、WebRTC 构建产物、codec/profile、输入像素格式，以及创建会话时的 specification 和关键 property。普通 HEVC 与 RTVC 必须视为两种不同会话分别探测。

创建会话成功不等于参数已经生效。能读取 effective value 时应回读；不能回读时，通过 encoder 标识、输出 telemetry 或可复现的行为差异确认。属性不支持、属性被忽略和会话创建失败应分别记录。

### 完成标准

- 能力结论带有完整的硬件、OS、产物和会话条件。
- 普通会话与 RTVC 会话的结果没有混用。
- 实验只纳入已经确认能够设置并生效的属性组合。
- 对未覆盖硬件或色彩格式明确标注未知，不把单机结果写成平台普遍事实。

### 项目证据

[`docs/experiments/2026-07-18-hevc-meeting.md`](experiments/2026-07-18-hevc-meeting.md) 记录了本机 VideoToolbox 能力探测及其适用条件。

## P06　实验矩阵必须直接回答业务选择

### 适用情况

比较 H.264/H.265、STATIC/ACTIVE MaxQP 或 VideoToolbox 特性开关，准备决定默认 codec 或参数时。

### 已经发生的问题

单独回答“HEVC 的某个 QP 应该设多少”容易把实验变成参数扫描，忽略会议投屏真正关心的静态文字、正文更新延迟、丢帧和码率。另一方面，H.264 与 H.265 使用相同 QP 数字，也不代表量化强度或主观画质相同；如果声称这种设置能够单独隔离 codec 差异，结论会超过对照本身的能力。

### 后续规则

实验按以下顺序组织：

1. 保留 H.264 作为当前业务参考组；
2. 固定内容、分辨率、帧率、码率目标、GOP 和 static-aware 状态切换方式，设置 H.264/H.265 编码参数尽可能对齐的直接对照；它用于观察同一业务约束下的端到端结果，不把相同 QP 解释成相同画质；
3. 比较 codec 效率时，使用相同码率预算下的画质和延迟，或者达到同一可接受画质时所需的码率，不能只比较 QP 数字；
4. 另设参考 WebRTC、VideoToolbox 实测和工程经验得到的 HEVC 候选区间；
5. 同时验证 STATIC 与 ACTIVE 的不同 MaxQP，不能只测单一常量；
6. 基础候选通过画质、真实内容延迟和稳定性准入条件后，才进入 Spatial Adaptive QP、Low-Latency Rate Control 或 B-frame 等扩展特性阶段。

每一阶段开始前写清它要改变哪项产品决策、主要指标和提前停止条件。新的组合只有在提出了新的、可证伪的假设时才增加。

### 完成标准

- 矩阵包含 H.264 业务参考、同业务约束下的跨 codec 直接对照和 HEVC 经验候选。
- static-aware 机制在 STATIC 与 ACTIVE 两种状态下都被验证。
- 相同 QP 只用于验证配置路径和观察结果，不被解释为等价画质或唯一 codec 差异。
- codec 效率结论来自相同码率预算或相同可接受画质下的对比。
- 扩展特性阶段由基础候选结果自动或明确门控，失败后不会继续组合爆炸。
- 报告最终回答“默认策略怎么选、为什么”，而不是只给出参数排行榜。

### 项目证据

[`docs/experiments/2026-07-18-hevc-meeting.md`](experiments/2026-07-18-hevc-meeting.md) 与 [`docs/experiments/2026-07-18-hevc-meeting-rerun.md`](experiments/2026-07-18-hevc-meeting-rerun.md) 记录了 H.264 参考、H.265 对照和 static-aware MaxQP 的实验设计。

## P07　固定实验内容，并验证发送端真正捕获的画面

### 适用情况

进行画质、正文滚动延迟、状态检测或 codec 对比实验时。

### 已经发生的问题

浏览器自动化设置的 `1920×1080` viewport（网页渲染区域）不等于 ScreenCaptureKit 实际捕获了同样的内容。地址栏、权限弹窗、桌面窗口、原生全屏、物理屏比例和宽高比不一致产生的黑边都会改变画面几何。延迟标记或质量裁剪区如果只按浏览器坐标设计，可能离开实际捕获区域，甚至采到桌面或黑边。

### 后续规则

正式实验使用本地固定文档、固定 Chrome 版本、固定 zoom、窗口或全屏模式、固定滚动距离与节奏。不要依赖在线文档、登录态页面或会自动变化的远程资源。

以发送端真实帧为准确定分辨率、有效内容区域、延迟标记区域和质量裁剪区。至少抽查发送端采集截图与 Android 最终画面，确认两侧看到的是同一段正文、同一滚动位置和可比较的有效区域。

主屏全屏实验还必须区分三种尺寸：Chrome 的页面 viewport 用于固定文档排版和滚动坐标；macOS 主屏使用 point 表示原生窗口几何，Retina 系统截图通常是其整数倍像素；ScreenCaptureKit 和编码器可以再把主屏缩放为 `1920×1080`。后两者都不能反向证明 Chrome 已经原生全屏。

本项目的 Chrome 全屏准入流程如下：

1. 使用独立 profile 启动实验专用 Chrome，并按 `--user-data-dir` 和 `--kiosk` 同时定位它的根进程，不能激活用户已有的其他 Chrome 窗口；
2. sender 已启动、但媒体测量尚未开始时，精确激活该进程，至少等待一秒让 macOS 完成前台切换，再通过原生 `CGEvent` 发送 `Control+Command+F`；
3. 从 `CGMainDisplayID` 读取真实主屏 bounds，从 `NSScreen.safeAreaInsets.top` 读取刘海屏顶部安全区，再从 `CGWindowListCopyWindowInfo` 读取该 Chrome 进程最大的 layer-0 窗口；只有窗口横向覆盖主屏、纵向覆盖顶部安全区以下至屏幕底部，才能发布 fullscreen-ready；
4. 页面 `innerWidth/innerHeight`、固定 zoom、初始滚动位置和 marker 几何仍需检查，但只作为页面契约，不能替代原生窗口准入；
5. fullscreen-ready 前存在自动化授权、权限、升级或其他系统弹窗时，本轮直接停止。清理弹窗后重新做短冒烟，不得把弹窗遮挡的画面纳入实验。

带刘海的 MacBook 原生全屏截图顶部可能保留与 `safeAreaInsets.top` 对应的黑色安全区。该黑条本身不是失败；地址栏、标签栏、菜单栏、Dock、桌面或其他应用窗口进入截图才是失败。实现或机器环境变化后，先保存并亲自查看系统级 `initial-static` 截图，再查看 sender 的真实采集帧；两张图都只包含固定文档和允许的顶部安全区后，才能开始正式 case。

### 完成标准

- fixture、HTML、浏览器、发送端程序和接收端程序的版本或 hash 已记录。
- 页面初始位置、每次滚动距离、间隔和总次数固定。
- 发送端原始帧证明没有地址栏、权限弹窗、锁屏、意外桌面或黑边污染目标区域。
- fullscreen-ready 同时包含真实主屏、顶部安全区和 Chrome 原生窗口 bounds；普通窗口即使页面 viewport 为 `1920×1080` 也会被拒绝。
- 延迟标记区域和质量裁剪区来自真实捕获几何，而不是只来自自动化 viewport。

### 项目证据

[`scripts/hevc_meeting_workload.py`](../scripts/hevc_meeting_workload.py) 和 [`scripts/test_hevc_meeting_workload.py`](../scripts/test_hevc_meeting_workload.py) 记录了固定文档与固定滚动负载。

## P08　正式批次前，从冷启动跑通一轮完整链路

### 适用情况

准备开始多轮正式实验，或者浏览器、发送端、接收端、relay、截图程序、构建产物或数据采集语义发生变化时。

### 已经发生的问题

实验曾因 HTTP 服务只在本机可达、Chrome 未处于目标状态、机器锁屏、Android emulator 消失、时钟校准缺失以及截图监听程序提前退出而失败。这些问题可能在单个进程的就绪检查中全部显示正常，却在真正的固定操作流程开始后暴露。

### 后续规则

第一轮正式数据前，从停止状态启动所有组件，完成一轮可丢弃的完整链路冒烟验证。运行前至少检查：

- console 已解锁，目标 Chrome 在前台并处于规定窗口或全屏状态；
- Chrome 精确版本与固定文档 hash 正确；
- Android 到 host 的真实 `/healthz` 请求返回成功，而不只是 host 自测端口；
- sender、receiver、signaling/relay 均达到可用状态；
- sender 与 receiver 的时钟校准完成；
- codec、QP、会话类型和构建产物身份符合本轮配置；
- 截图监听和日志采集的生命周期覆盖整个固定操作流程；
- 分析器能读取本轮产生的 telemetry 格式。

### 完成标准

- 冷启动冒烟轮完成了启动、规定次数的固定操作、接收端渲染、截图、日志收集和分析；当前会议文档实验为六次滚动。
- 冒烟轮的原始记录能证明延迟标记序号、状态切换和关键运行遥测都存在。
- 冒烟失败先修基础设施，不把不完整数据放入正式统计。
- 运行时组件、构建产物、采集内容或数据采集语义变化后重新执行冒烟轮。
- 只有分析器实现变化、原始数据格式和含义未变时，先用冻结的原始记录回放验证，不要求重跑真实链路。

### 项目证据

[`scripts/run-hevc-meeting-experiment.sh`](../scripts/run-hevc-meeting-experiment.sh)、[`scripts/run-android-tv-e2e.sh`](../scripts/run-android-tv-e2e.sh) 以及提交 `34268be`、`39e7a37`、`41df4fd`、`eb4a0d6` 记录了运行前检查和基础设施修复。

## P09　延迟标记必须与正文真实变化绑定

### 适用情况

使用可见延迟标记、事件序号或时间戳测量滚动、翻页、窗口切换等用户操作的端到端延迟时。

### 已经发生的问题

早期固定操作脚本先显示延迟标记（marker），再等待并执行正文滚动。接收端因此可能先看到标记，却还没有看到正文更新；测得的是叠加标记到达时间，不是用户真正看到内容变化的时间。标记自身也会触发 ScreenCaptureKit damage、静动态检测和编码，因而不是完全被动的观测手段。

### 后续规则

标记序号和时间戳必须与真实正文操作在同一个操作边界提交，中间不能插入等待。对于滚动操作，应在同一轮事件中更新标记并执行真实 `scroll`，而不是提前显示标记。标记在整个变化过程内必须留在发送端和接收端都可检测的区域，不能随正文立即滚出画面。

延迟至少拆成“操作到采集”和“采集到渲染”两段。只有带外事件编号能够在发送端和接收端与具体视频帧可靠关联时，才优先用它避免改变画面；普通本地日志或信令事件只能证明事件传播，不能证明对应画面已经完成采集和渲染。无法建立这种关联时仍需可见画面证据，并在报告中说明标记可能影响状态检测和编码行为。

### 完成标准

- 原始时间线能证明标记与正文操作之间没有预等待。
- 每个序号都能在发送端实际帧和接收端最终画面中配对。
- 抽查截图确认正文已真实更新，不只是标记数字变化。
- 指标名称准确描述被测事件，不把标记到达写成正文到达。
- 使用带外事件编号时，已验证它与两端具体视频帧的关联关系。

### 项目证据

[`scripts/hevc_meeting_workload.py`](../scripts/hevc_meeting_workload.py)、[`scripts/hevc_meeting_experiment.py`](../scripts/hevc_meeting_experiment.py) 及提交 `6c287e4`、`d829b1c`、`b8dc4d4` 记录了延迟标记与正文滚动的绑定修复。

## P10　静态状态机必须在没有新回调时继续推进

### 适用情况

状态变化依赖 quiet period、timeout 或“最后一次事件后等待一段时间”的 detector、debouncer 和 capture 状态机。

### 已经发生的问题

STATIC 的含义通常是事件源停止产生新帧。如果 quiet deadline 只在 capture callback 到来时检查，最后一帧之后没有新 callback，状态就永远不能进入 STATIC。即使使用 `asyncAfter`，若 timer 与高频 callback 共用同一条繁忙串行队列，也可能长期排在回调积压之后。

### 后续规则

activity 或 damage 只负责更新基于单调时钟的 deadline；独立且轻量的唤醒源负责在 deadline 到点时触发检查。状态检测器和 pixel buffer 仍回到原来负责状态写入的串行队列修改，避免并发状态分叉。系统只保留一个待执行唤醒任务，stop/restart 通过 generation 或等价令牌让旧任务失效。

只有在真实调度问题得到复现后才引入额外线程或复杂调度结构；不要为了理论上的边界制造多套状态所有权。

### 完成标准

- 单元测试覆盖“最后一次 damage 后再无 callback，仍按时进入 STATIC”。
- 压力验证覆盖 callback 队列积压，deadline 延迟有明确上界。
- stop/restart 后旧 timer 不会修改新一代状态。
- 状态写入仍由单一队列拥有，没有为防御罕见竞争增加重复状态。

### 项目证据

[`DamageIdleDetector.swift`](../apps/macos/WebRTCScreencast/Capture/DamageIdleDetector.swift)、[`DamageIdleDetectorTests.swift`](../apps/macos/WebRTCScreencastTests/DamageIdleDetectorTests.swift) 及提交 `d6859ce`、`557ac67` 记录了 deadline 调度修复。

## P11　系统重绘信息不能直接当作业务内容变化

### 适用情况

使用 ScreenCaptureKit `dirtyRects`、damage area 或其他 compositor metadata 判断画面是否动态时。

### 已经发生的问题

macOS 顶部 capture status strip 曾大约每秒产生一次近全宽、小高度的重绘，使 detector 无法进入 STATIC。`dirtyRects` 只表示 compositor 报告了重绘区域，不保证像素真的变化，也不保证变化属于用户关心的正文。反过来，按高度直接忽略顶部区域又可能吞掉真实网页 header、会议通知条或顶部 banner。

### 后续规则

先在真实设备采样噪声的形状、频率和像素变化，再设计窄范围候选。候选区域仍需进行像素核验；只有几何特征和像素结果都符合已知系统噪声时才忽略。metadata 缺失、无法解析、无法比较或结果不确定时，按动态内容处理，避免错误进入 STATIC 并收紧 MaxQP。

底层 ACTIVE/STATIC 状态切换总数不能代替业务操作覆盖。浏览器 compositor、延迟标记、截图、cursor 和系统 UI 都可能产生额外状态切换。每次固定滚动都应形成独立业务事件周期，并绑定本次 ACTIVE QP 生效以及随后 STATIC QP 恢复。

### 完成标准

- 过滤规则基于真实样本，并同时使用窄几何候选和像素核验。
- 测试同时覆盖系统顶部噪声和真实顶部业务内容，后者不会被忽略。
- 不确定路径保持 ACTIVE，并记录 fallback 次数与处理开销。
- 每次业务滚动都逐次完成状态往返；当前会议文档实验的六次滚动不能只用全局状态切换总数代替。

### 项目证据

[`DamageIdleDetector.swift`](../apps/macos/WebRTCScreencast/Capture/DamageIdleDetector.swift)、[`docs/experiments/2026-07-18-damage-idle-detector.md`](experiments/2026-07-18-damage-idle-detector.md) 及提交 `36c0368`、`ef87fc4` 记录了系统重绘噪声和业务 episode 的验证。

## P12　颜色问题按数值首次分歧逐层定位

### 适用情况

同一内容只在某个 codec、range 或设备上出现发白、压黑、浅灰丢失或亮度偏移时。

### 已经发生的问题

HEVC sender 到 Android 的静态文档曾明显发白。若直接改 QP、SPS/VUI、Android shader 或 renderer，很容易同时改变多个边界，无法知道真正根因。项目最终通过 codec 与输入 range 的正交对照，逐层比较数值，定位了 luma/color-range 边界。

### 后续规则

先建立 `codec × input range` 的最小正交矩阵，再按以下顺序寻找第一次数值分歧：

1. capture/input plane code value 与 color attachment；
2. 原生 VideoToolbox 编码后软件解码；
3. 实际 WebRTC encoder wrapper 和 packetization 后软件解码；
4. Android decoder Surface；
5. 生产使用的 `SurfaceViewRenderer` 最终画面。

“最小验证程序（probe）”只用于定位边界；真实生产 renderer 的端到端验证用于证明产品结果，两者不能互相替代。验证程序本身的资源路径、API 或构建失败，应先归为工具故障，不能算作颜色行为失败。

格式字段缺失不一定表示语义未知；规范可能定义默认值。默认语义应通过解析规则测试固定。长期回归同时保留故障格和修复格，不能只保留修复后的成功路径。

### 完成标准

- 正交矩阵一次只改变 codec 或 range，输入内容和其余链路保持一致。
- 报告指出首次出现数值分歧的具体边界，而不是只给最终截图结论。
- 软件解码、Android decoder Surface 和生产 renderer 的证据层级分开陈述。
- 自动指标与原始分辨率截图一致，修复没有通过改变亮度掩盖细节问题。

### 适用限制

本项目的 `420v + Rec.709` 结论只覆盖已验证的 8-bit SDR Rec.709 链路，不能外推到 HDR、Display P3、P010、物理 TV 的 vendor decoder 或 HDMI black level。

### 项目证据

[`docs/experiments/2026-07-18-hevc-color-range.md`](experiments/2026-07-18-hevc-color-range.md) 及提交 `1de5ce2`、`f58e812`、`3bf82c7`、`0eff712` 记录了分层定位与修复验证。

## P13　每次实验尝试必须可恢复、可归因，失败原始记录不能被覆盖

### 适用情况

正式实验包含多次运行、自动重试、断点恢复或多个外部进程时。

### 已经发生的问题

实验记录曾使用多行 JSON，却按 `wc -l` 计算尝试次数，导致恢复位置错误。另一些失败跨越 Chrome、发送端、Android 和分析器多个阶段，如果只记录“本轮失败”，就无法区分基础设施故障与真实业务失败。过早删除失败目录还会丢掉唯一可用于定位锁屏、就绪检查或测量问题的材料。

### 后续规则

每次尝试写入一个结构化记录，至少包含运行编号、尝试编号、最后成功阶段、失败阶段、直接错误、产物身份和最终分类。计数和恢复必须解析结构化对象，不能依赖文本行数。

只有已经明确分类为短暂基础设施故障的失败可以有限次自动重试。延迟标记丢失、正文未到达、业务状态往返不完整、画质或延迟超限属于实验结果，不能通过重试把失败样本覆盖掉。恢复时跳过已经标记为 `VALID` 的运行，从下一个未完成尝试继续。

测量语义、浏览器版本、构建产物或分析格式发生变化后，使用新的证据目录；不同语义的数据不能混入同一正式批次。失败和 `INVALID` 原始记录保留到报告定稿，必要时移到可恢复位置，而不是永久删除。

### 完成标准

- 尝试次数、状态和恢复位置来自结构化记录。
- 每个失败能归因到明确阶段，并区分基础设施失败与业务失败。
- 自动重试次数有上限，且不会重试掩盖业务失败。
- 语义变化后启用新的证据目录，旧记录仍可复核。

### 项目证据

[`scripts/run-hevc-meeting-experiment.sh`](../scripts/run-hevc-meeting-experiment.sh) 和提交 `34268be`、`39e7a37`、`41df4fd` 记录了分阶段归因、恢复和尝试计数修复。

## P14　原始记录、分析器和报告必须分层

### 适用情况

从 telemetry、截图或逐帧记录生成聚合指标、实验报告和产品建议时。

### 已经发生的问题

运行遥测字段从 `motion` 调整为 `active` 后，旧分析器继续读取旧字段，把实际完成的业务周期统计为零。另有聚合结果只保留每轮最大相邻帧间隔（frame gap），没有事件序号、起止时间和所属状态窗口，却被用于推断动态滚动期间发生画面冻结。聚合值没有保存的上下文，不能由报告补写出来。

### 后续规则

原始记录保持不可变；分析器修复后可以重新计算，但不能改写原始文件。每份正式结果同时记录实验执行脚本版本、分析器版本和数据格式版本。

任何结论都要能追溯到足够细的记录。延迟和相邻帧间隔至少保留事件序号、时间区间、STATIC/ACTIVE 状态和对应业务操作。若现有数据只支持“观察到较大的相邻帧间隔”，就只能这样陈述，不能进一步确定它发生在正文滚动期间。

参考组同样违反准入条件时，结论是当前方案或当前指标不足，不是“大家都失败所以可以通过”。需要重新检查门槛是否代表业务价值、测量是否正确，再决定是否重跑。

### 完成标准

- 原始记录不可变，重新分析结果存放在独立输出中。
- 正式报告记录执行脚本、分析器和数据格式版本。
- 每个关键结论都能定位到事件序号、时间区间、内容状态和截图或运行遥测。
- 数据不足时明确写“证据不足”，不进行超过数据粒度的因果归因。

### 项目证据

[`scripts/hevc_meeting_experiment.py`](../scripts/hevc_meeting_experiment.py) 及提交 `d829b1c`、`b8dc4d4` 记录了业务 episode 和分析格式修复。

## P15　机器准入、风险豁免和产品选择必须分别记录

### 适用情况

实验设置了自动准入条件，但业务负责人仍可能基于收益、已知风险或后续监控作出例外选择时。

### 已经发生的问题

某轮实验的机器判断仍为 `eligible=false`，随后基于业务价值和已知风险作出了继续推进的决定。如果报告把这类决定改写成“所有验收均已通过”，后续维护者将无法知道哪项风险仍然存在，也无法设计回滚条件。

这里的“机器准入”是指程序根据预先定义的画质、延迟和稳定性门槛自动给出的结果；“风险豁免”是指机器准入未通过，但业务负责人在明确适用范围、残余风险和回滚条件后作出的例外决定。它们不是同一个结论。

### 后续规则

报告分别保留三层信息：原始指标与机器准入结果、工程解释、最终产品选择。风险豁免不得修改原始 `eligible` 值；必须写明适用范围、未通过项、接受理由、监控信号和回滚条件。

实验阶段的提前停止也遵循同一原则。基础候选未达到门槛时，停止后续扩展特性组合；只有明确的新假设或测量缺陷得到修复后，才恢复实验。不能靠手工参数绕过阶段门控。

### 完成标准

- 报告明确区分“自动通过”“自动未通过但风险豁免”“拒绝推进”。
- 风险豁免不改写原始数据，并包含适用范围、监控和回滚条件。
- 后续扩展特性阶段有可执行的进入条件，失败时默认停止。
- 产品建议能追溯到机器结果和业务取舍，不能把取舍伪装成测量事实。

### 项目证据

[`docs/experiments/2026-07-18-damage-idle-detector.md`](experiments/2026-07-18-damage-idle-detector.md) 和 [`docs/experiments/2026-07-18-hevc-meeting-rerun.md`](experiments/2026-07-18-hevc-meeting-rerun.md) 展示了机器判断与最终建议的分层记录方式。

## P16　画质结论必须同时经过自动指标和人工原图检查

### 适用情况

准备比较 codec、MaxQP、颜色范围或最终 renderer 画质，并形成正式结论时。

### 已经发生的问题

SSIM-Y、PSNR-Y、QP histogram、VT drop 和 bitrate 能发现系统性差异，但可能漏掉文字边缘、浅灰边框、低饱和区域、block、ringing 和 ghosting。只看截图也可能被不同滚动位置、裁剪区、亮度范围或缩放方式误导。两类证据必须互相校验。

### 后续规则

每个正式实验组同时保留自动指标和代表性原图。原图至少覆盖发送端、Android 解码输出和最终 renderer 的初始、中间、结束状态。检查前先确认裁剪区、正文位置、叠加标记和滚动阶段可比较。

关键截图必须使用图片查看工具以原始分辨率亲自打开检查，不能只看 OCR、文件名、缩略图或分析器摘要。发现跨 codec 的系统性亮度或 range 差异时，先按 P12 修颜色链路，再比较 MaxQP；否则会把颜色错误误写成编码质量差异。

### 完成标准

- 自动指标与人工原图检查共同支持画质结论。
- 关键截图已用原始分辨率查看，且正文、滚动位置和裁剪区可比较。
- 文字边缘、浅灰元素、低饱和区域、块效应、振铃和拖影均有明确人工结论。
- 自动指标与原图冲突时先解释冲突，不选择性丢弃其中一类证据。

### 项目证据

[`docs/experiments/2026-07-18-hevc-meeting-rerun.md`](experiments/2026-07-18-hevc-meeting-rerun.md)、[`docs/experiments/2026-07-18-hevc-color-range.md`](experiments/2026-07-18-hevc-color-range.md) 及其关键截图记录了自动指标和人工原图检查。

## P17　发布实验材料前，单独完成隐私和产物身份检查

### 适用情况

准备把实验报告、截图、hash 或日志提交到仓库或发送给项目外人员时。

### 已经发生的问题

原始截图和日志会包含用户名、本机绝对路径、设备 UDID、网络诊断、TURN 配置和进程信息。它们对本地排障有价值，却不应随报告发布。另一个常见误差是 APK 或 executable 在每轮重新构建，却只记录一次 hash，并据此声称整批实验使用同一程序。

### 后续规则

原始实验目录保留在 Git 忽略的本地位置。可提交内容限于聚合结论、不可逆产物 hash、固定公开文档，以及重新编码并逐张检查过的关键截图。发布前检查可见文字、图片 metadata、文件名和日志片段；不能只依赖自动替换路径。

产物身份检查沿用 P03。若每轮重新构建 APK 或 executable，逐轮记录 hash；只有 hash 和实际运行记录都能证明一致时，才能声称整批使用同一产物。

### 完成标准

- 已检查并移除用户名、绝对路径、UDID、网络凭据、TURN 配置和无关进程信息。
- 提交到仓库的截图只包含固定公开文档和必要的实验标记。
- 每轮实际运行的 executable/APK 身份可核对；无法证明一致时不声称整批同源。
- 隐私清理只作用于发布副本，未破坏本地原始记录的可复核性。
